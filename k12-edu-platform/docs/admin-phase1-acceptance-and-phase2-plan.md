# 管理端一期验收与二期开发规划

## 一期验收整改结果

本轮优先处理一期阻塞项，目标是让管理端具备可构建、可编译、可继续演进的基础。

### 已完成

- 前端构建错误已修复：
  - `src/admin/views/system/SystemFeatureFlags.vue` 修复 `el-switch` 回调类型。
  - `src/admin/views/system/SystemUploadConfig.vue` 补齐 `loading` 状态并修正文案。
- 前端资源 API 类型已补充 `auditStatus`、`publishStatus`，保留旧 `status` 兼容字段。
- 后端新增 Maven Wrapper：
  - `mvnw.cmd`
  - `.mvn/wrapper/maven-wrapper.properties`
  - 固定 Maven `3.9.9`，避免本机 Maven `3.5.4` 无法运行编译插件。
- 后端资源状态新增双状态兼容骨架：
  - `audit_status`：草稿、待审核、审核通过、已驳回、复审中。
  - `publish_status`：未上架、已上架、已下架、定时上架、回收站。
  - 旧 `status` 暂时保留，用于兼容现有接口、搜索、前台展示与历史数据。
- 新增 SQL 迁移：
  - 后端 `sql/80_admin_resource_status_split.sql`
  - 同步回填 `audit_status`、`publish_status`
  - 新增 `v_admin_resource_main` 视图，作为统一资源审核域的过渡入口。
- 上传和管理端资源生命周期已同步写入新旧状态：
  - 保存草稿：`status=-1`，`audit_status=-1`，`publish_status=0`
  - 提交审核：`status=0`，`audit_status=0`，`publish_status=0`
  - 审核通过/发布：`status=1`，`audit_status=1`，`publish_status=1`
  - 驳回：`status=2`，`audit_status=2`，`publish_status=0`
  - 下架：`status=3`，`audit_status=1`，`publish_status=2`
  - 回收站：`status=4`，`audit_status=1`，`publish_status=4`

### 已验证

- 前端：`npm run build` 通过。
- 后端：`cmd /c mvnw.cmd -pl k12-resource -am -DskipTests compile` 通过。
- 说明：`clean compile` 时 `k12-common` 已重新编译通过，`k12-resource` 的 `clean` 被 Windows 文件锁阻止删除 `target`；随后使用非 clean 编译，`k12-resource` 已重新编译通过。

## 一期 Smoke Test 清单

### 路由与登录

- 未登录访问 `/admin` 应跳转 `/admin/login`。
- 普通用户访问 `/admin` 应进入 `/admin/403`。
- 管理员登录后应进入 `/admin/analytics/overview`。
- 后端 `/api/admin/**` 必须校验 `X-User-Id` 与管理员身份，不能只依赖前端路由守卫。

### 权限

- 菜单由后端 `sys_menu` 返回，前端侧边栏按权限展示。
- 按钮权限通过 `v-permission` 控制展示。
- 后端写接口必须有 `@RequiresPermission` 或服务内权限校验。
- 超级管理员可访问全部后台模块。
- 审核员只能访问审核相关模块和操作，不应获得系统配置、角色权限等能力。

### 资源审核

- 上传资源默认进入待审核或草稿，不允许默认直接前台发布。
- 待审队列只展示待审核资源。
- 审核通过必须记录审核日志。
- 审核驳回必须填写原因。
- 批量审核需要正确返回成功数和跳过原因。
- 重复资源、敏感词、文件安全状态应在审核预览中可见。

### 上传与草稿

- 保存草稿后应在用户个人中心“我的上传/草稿箱”可见。
- 草稿不进入前台搜索、推荐和下载。
- 草稿提交审核后进入管理端待审队列。
- 驳回资源应允许查看驳回原因，并可复制为新草稿重新提交。

### 日志与追责

- 审核、驳回、上下架、删除、恢复、推荐、置顶、角色权限、系统配置修改都要写操作日志。
- 日志应包含操作者、接口、模块、动作、权限、状态、耗时、错误信息。

### 数据字典与目录

- 上传页的学段、年级、学科、版本、册别、栏目、单元、资源类型必须来自统一 taxonomy/catalog/dictionary 数据源。
- 小学阶段只能展示一年级到六年级，不能出现七八九年级。
- 单元目录不应前端写死，必须按学段、学科、版本、册别联动加载。

## 二期开发规划

二期不包含会员、订单、支付、会员权益等内容。本系统定位为内部成员使用，权限与资源治理优先级高于商业会员能力。

### 第一步：一期验收固化

目标：把一期从“能跑”固化为“可验收”。

- 整理乱码文件，统一 UTF-8 无 BOM。
- 给前端增加管理端 smoke test 脚本或最小 E2E 用例。
- 给后端增加 admin 权限拦截、审核、资源生命周期的单元/集成测试。
- 将 SQL 迁移纳入统一执行顺序。
- 整理部署说明：必须使用 `mvnw.cmd`，不再依赖本机 Maven。

### 第二步：上传资源业务标准化

目标：前台上传形成完整闭环。

- 上传入口支持保存草稿、提交审核、撤回草稿、删除草稿。
- 个人中心增加：
  - 我的草稿
  - 待审核
  - 已发布
  - 已驳回
  - 已下架
- 驳回资源显示驳回原因、审核人、审核时间。
- 驳回资源可复制为草稿并重新提交。
- 上传成功后不直接叫“发布资源”，应改为“提交审核”。

### 第三步：统一分类、目录、字典源

目标：解决学段、年级、单元、类型错乱问题。

- 上传页、资源列表、搜索筛选、管理端分类目录共用一套接口。
- 学段和年级强绑定：
  - 小学：一年级至六年级
  - 初中：七年级至九年级
  - 高中：高一至高三
- 教材版本、册别、单元、课时按级联关系加载。
- 资源类型、资源形态、教学场景、地区、年份、考试类型改为后台字典配置。
- 禁止新增前端硬编码分类常量。

### 第四步：统一资源审核域

目标：从 `oss_primary_chinese_resource` 过渡到平台级资源治理。

- 先使用 `v_admin_resource_main` 作为统一管理视图。
- 后续新增 `resource_main` 聚合表或标准主表。
- 统一字段：
  - source_type
  - stage
  - grade
  - subject
  - edition
  - volume
  - catalog_node_id
  - resource_type
  - file_info
  - uploader_id
  - audit_status
  - publish_status
  - visibility
  - source
  - copyright_status
- 管理端审核中心按统一资源域展示，而不是绑定单一小学语文表。

### 第五步：权限与追责强化

目标：符合内部平台、行业运营和后期测评要求。

- RBAC 保留，继续强化数据权限。
- 数据权限增加：
  - 全部数据
  - 本人上传
  - 指定学段
  - 指定学科
  - 指定地区
  - 指定组织
- 导出用户数据、资源数据单独权限控制。
- 系统配置、角色权限、数据导出必须二次确认并记录日志。
- 关键日志不可物理删除，只能归档。

### 第六步：资源质量治理

目标：审核中心从“人工通过/驳回”升级为内容质量控制台。

- 敏感词词库管理。
- 重复资源检测和相似度评分。
- 文件安全状态：待检查、安全、风险、未知。
- 预览失败队列。
- 低质资源标记。
- 版权/来源字段。
- 驳回原因统计。
- 审核员工作量统计。
- 审核 SLA：待审核时长、平均审核时长、超时队列。

### 第七步：运营和测评能力

目标：支撑真实运营、内部管理和后期测评材料。

- 用户增长、活跃、上传、下载、收藏、浏览统计。
- 资源增长、审核通过率、驳回率、下架率。
- 各学段、学科、版本、类型资源分布。
- 搜索无结果词、热词、点击转化。
- 热门资源、低质量资源、长期无人访问资源。
- 违规操作追踪。
- 管理端首页形成运营大盘。

## 行业准则落地原则

- 前台展示的数据尽量来自管理端配置，不能长期写死在前端。
- 所有关键操作必须可追责。
- 所有删除先软删除或进回收站。
- 审核和发布要分离。
- 分类目录必须统一来源。
- 文件上传必须有格式、大小、安全状态控制。
- 用户数据导出必须最小权限。
- 管理端不能依赖前端权限，后端必须强校验。
- 二期不做会员、订单、支付、会员权益。
